# Machine Learning Research Guide

The Machine Learning (ML) Research Layer is a comprehensive engine integrated inside the `quant-engine` framework. It allows quantitative researchers to engineer features, calculate target variables, perform Bayesian hyperparameter sweeps, run walk-forward validation, analyze model explainability, persist trained models, and execute **Hybrid Strategy + ML Research** via the **Trade Quality Classifier (TQC)**.

---

## Architecture Overview

The ML Layer resides under the `src/quant_engine/ml/` package and is structured cleanly around the single responsibility principle:

* **`datasets/`**: Compiles raw OHLCV and trade history data into aligned feature matrices and labels, dropping warmup periods and handling timezones.
* **`features/`**: Computes multi-category feature vectors (technical, candle metrics, volatility, volume, time patterns, session periods, overnight gaps, and rolling stats).
* **`labels/`**: Generates machine learning targets including binary direction, three-class signals, regression returns, and meta-labeling.
* **`models/`**: Defines the unified `BaseModel` adapter interface and provides interchangeable wrappers for XGBoost, LightGBM, CatBoost, RandomForest, and ExtraTrees.
* **`tuning/`**: Integrates Optuna for single and multi-objective hyperparameter tuning using cross-validation splits.
* **`explainability/`**: Generates feature importances, permutation importances, SHAP summary charts, and markdown top-feature reports.
* **`persistence/`**: Saves, loads, versions, and tracks metadata (metrics, feature lists, parameter configurations) for trained models.
* **`metrics/`**: Computes standard classification and regression evaluation scores.
* **`pipelines/`**: Sequences the standard model training flow (`training_pipeline`) and the trade filtering workflow (`hybrid_pipeline`).

## Configuration Reference

You can configure the ML layer in your YAML research configuration file. It fits natively into the existing structure:

```yaml
machine_learning:
  enabled: true                       # Set to true to execute the ML stage

  dataset:
    source: generated_trades          # "ohlcv" (standard training) or "generated_trades" (hybrid trade classifier)
    features:
      technical: true                 # Compute technical indicators (EMA, RSI, MACD, etc.)
      candle: true                    # Compute candle body and shadow ranges
      volatility: true                # Compute normalized ATR, rolling standard dev, BBands width
      volume: true                    # Compute volume log returns, volume ratios, OBV
      time: true                      # Compute hour of day, day of week, month
      session: true                   # Compute Indian market session flags (morning, midday, afternoon)
      gap: true                       # Compute overnight gap returns
      rolling: true                   # Compute rolling close/volume statistics
      rolling_windows: [5, 10, 20, 50] # Windows to use for rolling statistics

  labels:
    type: binary                      # binary, three_class, regression, or meta_labeling
    future_horizon: 10                # Future bars ahead to compute target variables
    threshold: 0.005                  # Threshold return (e.g. 0.5% return) for direction changes

  models:
    - lightgbm                        # Models to train/test
    - xgboost
    - random_forest
    - extra_trees
    - catboost

  tuning:
    enabled: true                     # Optimize model hyperparameters
    trials: 50                        # Number of Optuna trials
    cv_folds: 3                       # Time-series cross-validation folds
    objectives:                       # Optimization targets
      - accuracy

  explainability:
    shap: true                        # Generate SHAP summary plots
    permutation_importance: true      # Calculate permutation feature rankings
    feature_importance: true          # Log native model feature importances

  deployment:
    confidence_threshold: 0.80        # Confidence probability to accept a filtered trade entry
```

## Walk-Forward ML Validation

To avoid lookahead leakage and overfitting, the ML layer enforces strict chronological walk-forward validations:
1. Data is sorted chronologically.
2. For standard runs: the dataset is split into `Train` (e.g. first 70%), `Validate` (next 15%), and `Test` (last 15%).
3. During tuning, the Train/Validate portion is evaluated using a time-series cross-validation split (expanding folds) to simulate walk-forward optimizations.
4. The final model is fitted on the Train/Validate sets and evaluated on the untouched out-of-sample `Test` set.

## Hybrid Strategy + ML (Trade Quality Classifier)

This workflow evaluates signals generated by rule-based strategy genomes:

```
Generated Strategy ──> Trade Candidate ──> Feature Extraction ──> ML Confidence ──> Accept/Reject
```

1. The orchestrator runs strategy discoverers and backtesters on historical OHLCV data.
2. For top strategies, the trade logs (entries and exits) are collected.
3. At the exact timestamp of each trade candidate's entry, a feature vector is extracted.
4. The label is binary: `1` if the trade's actual backtest PnL was positive, `0` otherwise (meta-labeling).
5. The dataset (consisting only of trade entry rows) is split chronologically into 70% Train, 30% Test.
6. A Trade Quality Classifier is trained to learn the market states under which the strategy succeeds.
7. During the OOS test window, trades are accepted only if the classifier's predicted probability of success $\ge$ `confidence_threshold`.
8. The filtered trades are evaluated, and comparison metrics are saved to `hybrid_ml_comparison.csv`.

## How to Extend the ML Layer

### 1. Adding a New Feature Category
To calculate new custom features:
1. Open [extractor.py](file:///root/openalgo-autonomous-research/src/quant_engine/ml/features/extractor.py).
2. Define a configuration toggle flag in `__init__`.
3. Add the calculation logic inside `extract_features` aligning it with the timestamp index of the DataFrame.

### 2. Adding a New Model Adapter
To integrate a new ML model (e.g. an LSTM or SVM):
1. Open [base.py](file:///root/openalgo-autonomous-research/src/quant_engine/ml/models/base.py) to review the `BaseModel` abstract class interface.
2. Open [adapters.py](file:///root/openalgo-autonomous-research/src/quant_engine/ml/models/adapters.py).
3. Implement your new adapter class inheriting from `BaseModel` implementing `train`, `predict`, `predict_proba`, `save`, and `load` methods.
4. Register the new adapter string key inside `get_model_adapter` factory function.
